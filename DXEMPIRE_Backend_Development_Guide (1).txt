1. Overview & tech stack

This is the complete step-by-step backend development guide for the DXEMPIRE platform. It covers every module, database table, API endpoint, service class, and integration — from project bootstrap to production deployment. Follow each section in order; later modules depend on earlier ones.

2.3 Route structure (routes/api.php)

  All routes are prefixed with /api/v1. Group routes by module with middleware applied at group level — not on individual routes.

Route::prefix('v1')->group(function () {

  // Public routes (no auth)

  Route::post('/auth/send-otp', [AuthController::class, 'sendOtp']);

  Route::post('/auth/verify-otp', [AuthController::class, 'verifyOtp']);

  // Authenticated routes

  Route::middleware('auth:sanctum')->group(function () {

    Route::get('/auth/me', [AuthController::class, 'me']);

    Route::post('/auth/logout', [AuthController::class, 'logout']);

    // Super admin only

    Route::middleware('role:super_admin')->prefix('admin')->group(function () {

      Route::apiResource('users', UserController::class);

      Route::apiResource('partners', PartnerController::class);

      Route::put('settings/logistics', [SettingsController::class, 'logistics']);

    });

    // Procurement module

    Route::middleware('role:super_admin  warehouse_staff')->prefix('procurement')->group(function () {
      Route::apiResource('suppliers', SupplierController::class);

      Route::post('receive', [ReceivingController::class, 'store']);

    });

    // ... (repeat pattern for every module)

  });

});

4. Phase 1 — Authentication & RBAC

  Phase 1              Auth, OTP login, roles & permissions Build this first — every other module depends on auth middleware Days 9–13
4.1 OTP login flow

  #                    Task                      Detail
  1                    Send OTP endpoint         POST /api/v1/auth/send-otp. Accepts { phone }. Validates phone format. Generates 6-digit OTP. Hashes with bcrypt before storing in otp_codes. Deletes any previous unused OTP for that phone. Dispatches SendOtpJob (queued) → calls MSG91 API. Returns { message: "OTP sent" } — never return the OTP itself.
  2                    Verify OTP endpoint       POST /api/v1/auth/verify-otp. Accepts { phone, code }. Finds latest unexpired OTP for phone. Checks Hash::check($code, $otp->code). If valid: marks verified_at, finds/creates User, creates Sanctum token, registers Expo push token if provided. Returns { token, user { id, name, role } }.
  3                    Token structure           Sanctum personal access token. Token stored in personal_access_tokens table. Include role in token name for quick identification. Set token expiry to 30 days. For mobile: long-lived token. For web: also 30 days (user explicitly logs out).
  4                    Auth middleware           auth:sanctum middleware on all protected routes. Returns 401 with { message: "Unauthenticated" } if token missing or expired.
  5                    Logout                    POST /api/v1/auth/logout. Calls $request->user()->currentAccessToken()->delete(). Returns { message: "Logged out" }.
  6                    Get current user          GET /api/v1/auth/me. Returns { id, name, phone, role, permissions[] }. Frontend uses this on app load to restore session.
4.2 Roles & permissions (Spatie)

Create all roles and permissions in a seeder — never hard-code them in controllers.

// database/seeders/RolesPermissionsSeeder.php

$roles = ['super_admin','sales','warehouse_staff','qc_engineer','packing_staff','accounts','hr_manager','b2b_partner'];

foreach ($roles as $role) { Role::create(['name' => $role, 'guard_name' => 'api']); }

// Permissions (module.action format)

$permissions = [

  'users.view','users.create','users.edit','users.delete',

  'inventory.view','inventory.edit',

  'orders.view','orders.approve','orders.dispatch',

  'finance.view','finance.edit',

  'hr.view','hr.edit','payroll.run',

  'analytics.view','settings.edit',

];

foreach ($permissions as $p) { Permission::create(['name' => $p, 'guard_name' => 'api']); }

// Assign permissions to roles

Role::findByName('super_admin','api')->givePermissionTo(Permission::all());

Role::findByName('sales','api')->givePermissionTo(['orders.view','orders.approve','inventory.view','analytics.view']);

Role::findByName('warehouse_staff','api')->givePermissionTo(['inventory.view','inventory.edit','orders.view','orders.dispatch']);

4.3 Role middleware

// app/Http/Middleware/RoleMiddleware.php

public function handle(Request $request, Closure $next, ...$roles): Response {

  if (! $request->user()->hasAnyRole($roles)) {

    return response()->json(['message' => 'Forbidden'], 403);

  }

  return $next($request);

}

// Register in bootstrap/app.php: ->withMiddleware(fn($m) => $m->alias(['role' => RoleMiddleware::class]))

5. Phase 2 — Procurement & sourcing

5.1 Module logic

  #                    Task                      Detail
  1                    Supplier management       CRUD for suppliers. Types: dealer, importer, buyback_partner. GST number stored and validated format. Supplier balance tracking (what we owe them).
  2                    Purchase orders           Create PO against a supplier with expected items. Status: draft → placed → received. PO stores expected count, actual received count.

  6                    After receiving           Fire ProductReceived event → listener creates QC task → listener sends notification to QC engineer mobile app.

6. Phase 3 — Quality control (QC)

  Phase 3              QC workflow, grading S1–S5, refurbishment tracking Products flow through QC before entering sellable inventory Days 22–27
6.1 QC flow logic

  #                    Task                      Detail
  1                    QC queue                  GET /qc/pending returns all products with status=received, ordered by received_at ASC. QC engineer picks from top.
  2                    Grade submission          POST /qc/grade. Body: { product_id, grade, condition_notes, outcome }. Outcome: pass │ repair │ reject. Validates QC engineer role.
  3                    Pass path                 Creates qc_record. Updates product: grade=S1–S5, status=in_stock, qc_passed_at=now. Calculates and sets selling_price based on grade rules in settings table. Fires StockAdded event.
  4                    Repair path               Creates qc_record with outcome=repair. Updates product status=refurbishment. Creates refurbishment_job record. Sends notification to repair team.
  5                    Reject path               Creates qc_record with outcome=reject. Updates product status=rejected. Flags product against supplier for return. Sends notification to procurement team.
  6                    Grade rules               Grade pricing rules stored in settings table as JSON: { S1: 0.85, S2: 0.75, S3: 0.65, S4: 0.50, S5: 0.30 } — multipliers of purchase price. Admin can update from dashboard.
  7                    Refurbishment done        PUT /qc/refurbishment/{id}/complete. Re-enters QC cycle: product status → received, new qc_record created.

7. Phase 4 — Inventory management

  Phase 4              Stock engine, bin/rack, real-time counts, alerts Central inventory that all other modules read from Days 28–33
7.1 Inventory module logic

  #                    Task                      Detail
  1                    Stock query engine        GET /inventory accepts filters: category, grade, status, bin_id, brand, search(IMEI/serial). Use Eloquent scopes for each filter. Paginate at 50 per page. Always eager-load: bin, qcRecord, supplier.
  2                    Real-time counts          Do not cache counts — query live. Stock count = products.count() where status=in_stock, grouped by category. Run in <100ms with proper indexes on status and category columns.
  3                    Bin management            Bins have a capacity. Before moving product to bin: check bin.current_count < bin.capacity. On move: create bin_movement record, update product.bin_id, update bin.current_count (±1).
  4                    Low stock alerts          Laravel scheduled job runs every 30 minutes. Queries: SELECT category, COUNT() FROM products WHERE status=in_stock GROUP BY category. If count < threshold (stored in settings), fire LowStockAlert event → push notification to admin.
  5                    Stock availability        GET /inventory/availability used by B2B partner app and CRM. Returns { phones: { S1: 12, S2: 45 ... }, laptops: {...}, accessories: {...} }. Cache this for 60 seconds (Redis) since it is called frequently.
  6                    Product detail            GET /inventory/{id} returns full product history: qc records, bin movements, who received it, purchase price, current selling price.
  7                    Bulk export               GET /inventory/export returns Excel file via Laravel Excel. Applies same filters as list endpoint. Stream response to avoid memory issues.

8.1 CRM module logic

  #                    Task                      Detail

9. Phase 6 — Order processing

  Phase 6              Full order lifecycle — create to delivery Most complex module — 6 status stages, payment verification, logistics integration Days 41–50
9.1 Order lifecycle states

  pending: Order created. Payment not yet confirmed. Stock not locked.
  approved: Payment verified OR credit approved. Stock locked (cannot be sold to anyone else). Warehouse pick request created.
  picking: Warehouse staff assigned and started picking items.
  packing: All items picked. Packing team has the order. IMEI scan confirms each item.
  dispatched: Handed to courier. AWB number attached. Tracking active.
  delivered: Courier confirms delivery. GST invoice final. Credit updated for dealer.
  cancelled: Cancelled before dispatch. Stock unlocked. Payment refunded if paid.
  returned: Returned after delivery. Stock enters QC cycle. Credit note issued.
9.2 Order creation logic

  #                    Task                      Detail
  1                    Receive order             POST /orders. Accepts { dealer_id OR customer_id, items: [{ product_id }] }. Validate: all products exist, status=in_stock, not already in another order.
  2                    Stock validation          In a DB transaction: lock products rows with SELECT FOR UPDATE. Check each product.status == in_stock. If any product unavailable: rollback + return 422.
  3                    Credit validation         If dealer order: check credit_used + total <= credit_limit. If fails: return 422 with available credit amount.
  4                    Calculate totals          For each item: unit_price = product.selling_price. Apply dealer price tier discount. Calculate GST: if dealer is GST registered use GSTIN, apply 18% GST. Sum totals.
  5                    Create records            Create order record. Create order_items records (one per product). Create Razorpay payment order via RazorpayService. Return { order, razorpay_order_id } to frontend.
  6                    Generate order number     Format: DX-YYYY-NNNNN. Use DB sequence or: SELECT MAX(id) + 1 padded to 5 digits. Must be unique.
9.3 Payment and approval logic

  #                    Task                      Detail
  1                    Razorpay webhook          POST /webhooks/razorpay (public, no auth). Verify webhook signature using HMAC SHA256 with Razorpay webhook secret. On payment.captured event: find order by razorpay_order_id, update payment status=paid, call OrderService::approve($order).
  2                    OrderService::approve     Wraps in DB transaction. Updates order status=approved. Updates each product status=sold. Reduces dealer credit available. Creates warehouse pick request. Fires OrderApproved event.
  3                    OrderApproved event       Listeners: (a) Send push notification to warehouse staff app, (b) Send WhatsApp message to dealer, (c) Log to audit_log.
  4                    Credit order approval     For dealers using credit (no upfront payment): sales rep manually calls PUT /orders/{id}/approve after verifying. Same OrderService::approve() runs.
  5                    Manual approval           Super admin and sales role can manually approve any pending order. Useful for cash payments or bank transfers confirmed offline.
9.4 Dispatch logic

  #                    Task                      Detail
  1                    Packing confirmation      PUT /orders/{id}/packing-complete. Packing staff scans each product IMEI to confirm. Validates: scanned IMEIs match order_items. Updates status=packing.
  2                    Generate AWB              POST /orders/{id}/create-shipment. Calls LogisticsService::createShipment(). Logistics adapter calls active provider API (DTDC / Delhivery etc). Stores AWB number and tracking URL on order.
  3                    Mark dispatched           PUT /orders/{id}/dispatch. Updates status=dispatched, dispatched_at=now. Fires OrderDispatched event → sends tracking details to dealer via WhatsApp + push notification.
  4                    Delivery confirmation     Logistics providers send webhook on delivery. Or staff manually marks delivered. Updates status=delivered, delivered_at=now. Triggers invoice PDF generation + delivery notification.
  5                    Returns                   PUT /orders/{id}/return. Validates order is delivered. Creates return record. Updates order status=returned. Each product status=received (re-enters QC). Issues credit note to dealer.

// app/Integrations/Logistics/DtdcDriver.php  implements LogisticsInterface

// app/Integrations/Logistics/DelhiveryDriver.php  implements LogisticsInterface

// app/Integrations/Logistics/ShiprocketDriver.php  implements LogisticsInterface

// app/Providers/LogisticsServiceProvider.php

$this->app->bind(LogisticsInterface::class, function () {

  $provider = Setting::get("logistics_provider"); // "dtdc"   "delhivery": etc
  return match($provider) {

    "dtdc"       => new DtdcDriver(),

    "delhivery"  => new DelhiveryDriver(),

    "shiprocket" => new ShiprocketDriver(),

    default      => throw new InvalidArgumentException("Unknown provider: $provider"),

  };

});

// Inject LogisticsInterface in controllers — never call a driver directly

14. API standards & response format

14.1 Universal response format

Every API response — success or error — follows this structure. Never return raw model data without wrapping.

// SUCCESS response

{

  "success": true,

  "message": "Order approved successfully",

  "data": { ... },          // null for actions with no data

  "meta": {                 // only on paginated responses

    "current_page": 1,

    "per_page": 50,

    "total": 423,

    "last_page": 9

  }

}

// ERROR response

{

  "success": false,

  "message": "Validation failed",

  "errors": {               // only for 422

    "phone": ["The phone field is required."],

    "grade": ["Invalid grade value."]

  }

}

14.2 HTTP status codes used

  200 OK: Successful GET, PUT, PATCH — returns data
  201 Created: Successful POST that creates a resource — returns created object
  204 No Content: Successful DELETE — no body returned
  400 Bad Request: Malformed request — missing required field, wrong format
  401 Unauthorized: No token or expired token — must login again
  403 Forbidden: Valid token but wrong role for this endpoint
  404 Not Found: Resource does not exist — or soft-deleted (treat same)
  422 Unprocessable: Validation failed — return errors object with field-level messages
  429 Too Many Requests: Rate limit hit — implemented via Laravel Throttle middleware
  500 Server Error: Unexpected error — logged to Sentry, generic message returned to client
14.3 Pagination standard

GET /inventory?page=1&per_page=50&sort=created_at&direction=desc
